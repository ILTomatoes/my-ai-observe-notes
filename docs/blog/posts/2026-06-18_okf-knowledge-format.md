# 谷歌 OKF 和 AI 知识管理的碎片化困境

谷歌今天发布了一个叫 Open Knowledge Format（OKF）的开放规范，要解决的问题很明确：AI agent 需要的上下文知识散落在各种地方，没有通用格式，没有互通标准。读完规范和配套工具后，我有几点观察。

## 我的观察

### AI 的两根支柱：知识和推理

大模型的能力可以拆成两个部分：**知识**和**推理**。推理能力在持续进化，但知识始终是个瓶颈——模型内的世界知识是静态的，不可能每时每秒都在训练。而真正让 agent 在实际场景中发挥作用的，恰恰是那些动态的、不断变化的上下文知识：静态的领域知识、动态的行动记录、反思、学会的经验。

怎么把这些知识实时交给模型？目前的做法是**让 agent 动态组织上下文**，而载体大量依赖 Markdown 文件——Obsidian 知识库、AGENTS.md、CLAUDE.md 这类约定文件、数据团队的元数据仓库。

### 碎片化才是真正的痛点

问题不在于 Markdown 本身不好用，而在于**每家做法都是定制的**。Karpathy 的知识库和你团队的知识库和某家厂商导出的目录，长得像——都是 Markdown、都有 frontmatter、都有交叉链接——但它们没有被设计成可以互通的。没有人约定每个文档应该有哪些字段，也没有人约定文件名代表什么含义。

知识还是被锁在各自的团队里，每次搭新 agent 都得重新造轮子。

### OKF 的答案：一个格式，不是又一个平台

谷歌发布的 OKF 是一个**格式规范**，不是服务，不需要 SDK，不绑定任何云平台。核心设计很简单：

- 一个 OKF bundle 就是一个 Markdown 文件目录
- 每个文件代表一个概念（数据表、数据集、指标、操作手册、API）
- 文件路径就是这个概念的唯一标识
- 每个文件：YAML frontmatter（结构化字段）+ Markdown 正文（自由内容）
- 概念之间用 Markdown 链接互相引用，整个目录就是一张关系图

唯一的强制约束：**必须有 `type` 字段**。其他全自由。

### 假如所有 agent 都用同一种知识规范？

这是最值得想的问题。如果真的有一种统一的知识规范，知识就变得**可评估、易于管理、可分享、可积累**。agent 之间的知识不再是孤岛，一个 agent 生成的知识文件可以被另一个 agent 直接消费。

但现实是：OKF 目前更偏向**静态元数据目录**（数据表 schema、API 文档），而不是 agent 的运行时记忆和经验积累。这块可能需要在 OKF 之上再加一层。

## AI 锐评

### 你漏了一层：ECC 这类项目就是在解决同一个问题

你提到"很多依赖于 Markdown"，但没有展开对比。实际上，像 ECC（Everything Claude Code）这样的 AI 编码 agent 框架，其 rules/、skills/、agents/ 目录本质上就是一个**没有标准化的知识库**。每个 rule 文件有 YAML frontmatter，有 "When to Activate" 声明，有交叉引用——和 OKF 的设计理念高度重合。

区别在于：ECC 的格式是自定义的，和 Cursor、Gemini、OpenCode 的知识格式互不兼容。如果用 OKF 规范重写，理论上就能被其他 agent 框架直接消费。**标准化的价值不在于格式本身多好，而在于互通带来的网络效应。**

### "让 LLM 自己更新知识库"——理想很美，现实很骨感

Karpathy 说"LLM 不会感到无聊，不会忘记更新交叉引用"，这话对了一半。LLM 确实擅长批量更新结构化文件，但**没有人类审核闭环的话，知识库会快速退化**。agent 生成的知识文件质量参差不齐，错误会累积，过时信息会堆积。

OKF 的 `type` 字段强制约束是最小的一步，但**知识新鲜度的保障机制**才是真正的难题。格式规范解决的是"怎么存"，不是"存的对不对"。

### 动态知识才是 OKF 的盲区

OKF 设计得很优雅，但它覆盖的主要是**静态元数据**：数据表的 schema、API 的文档、指标的定义。真正让 agent 变聪明的动态知识——这次任务的失败原因、这个用户的偏好、上次踩过的坑——目前没有被 OKF 的设计语言很好地表达。

如果你要给 agent 建一个经验库，需要的不只是 `type: table` 这样的分类标签，还需要时间戳、置信度、关联的执行记录、衰减机制。这些在 OKF v0.1 里看不到。

### 三个设计原则里，"最小约束"最值得学

OKF 对每个文件只强制要求一件事：必须有 `type` 字段。其他字段、文件体的内容和结构，全由使用者决定。这种**"管互通的边界，不管内容本身"**的设计哲学，比很多试图定义完整 schema 的规范聪明得多。

约束越少，采纳门槛越低，网络效应起来得越快。这是 OKF 最值得关注的设计决策。
