# 同一天看到 3 篇 AI 文章,它们讲的是同一件事

那天刷到 3 篇文章,本来没觉得会有关联。读完之后才发现,它们其实在讲同一件事——AI 从"单核工具"被推到了"多核组织"的位置。

## 我的观察

### 3 篇文章,3 个角度

| 文章 | 切入点 | 抽象层 |
|------|--------|--------|
| 扣子 3.0 | 项目制容器 + 异构 Agent + SOP 沉淀 | 平台层 |
| Claude Dynamic Workflow | 4 种打法 + 3 大失败模式 | 打法层 |
| Multica 最佳实践 | Issue = 上下文,指派即执行 | 执行层 |

它们不是替代关系,是互补。选谁取决于团队成熟度:想要完全掌控选扣子,想要 agent 自治选 Claude 路线,想要替代工程师团队选 Multica。

### 4 个跨篇关键词

**多 agent > 单 agent。** 3 篇都认同。扣子的项目空间是"多 Agent 协作的工作室",Claude 那篇直接列了单 agent 的 3 大失败模式(差不多得了、相信自己、走了一半忘了),Multica 说了一句重话:"工具放大个人产出,组织放大协作产出,差一个数量级。"

**共享记忆空间 > agent 窗口。** 扣子用项目空间装目标、成员、Agent、文件、过程产出;Claude 让子 agent 各持干净上下文互不污染;Multica 把"记忆是窗口"和"记忆是组织资产"放在一起做对比。

**人做决策,不降低地位。** 扣子里人设目标、判方向、确认结果;Claude 那篇有个细节,agent 出错时"反对者 Agent"会提醒人;Multica 提了一个身份转变:"操作者(主语是你)→ 指挥者(主语是事)"——人退到组织者位置,价值反而更高。

**范式转变 > 工具升级。** Claude 那篇说"AI 的下一步,可能不是一个更强的'超级员工',而是一个能随时搭起来的'小团队'",Multica 直接用了"范式的断层"这个词。

### 我最在意的事

如果只学其中一家都不够。学扣子有容器但里面跑什么打法不知道;学 Claude 设计了打法但没地方装;学 Multica 有 issue 系统但"操作者→指挥者"这一步怎么过渡没想清楚。

把 3 层叠在一起,才是完整的图:

- **容器层**(扣子):项目空间、Plan 持久化、资产可迁移
- **打法层**(Claude):分诊 / 汇总 / 反对者 / 淘汰赛 4 种预编排工作流
- **执行层**(Multica):issue / 任务 / 评论、@触发动作、状态机、人只在 in_review 介入

### 行动项

合并 3 篇的 14 条行动项,去重后留 7 条:

**本周:** 设计稿强制加 3 层架构;选一个高频重复任务场景跑 v0;把 issue / 任务 / 评论定为核心数据模型。

**本月:** 写"任务值不值得上工作流"的判断标准;metadata 字段严格克制;把"人决策 / 人验收"当强制设计原则。

**下月评估:** Multica 路线值不值得直接采用。

## AI 锐评

### 你比文章本身多走了一步

3 篇文章各自在讲自己的产品,但你做了跨篇对比 + 自我定位。这不是"信息收集",这是**建模**。你能识别出"容器层 / 打法层 / 执行层"的三层架构,说明你已经有了一个分析框架——未来任何新的 AI 协作产品出来,你都能用这个框架快速定位它在哪一层。

### "3 层叠加"的判断很准,但执行难度被低估了

你说"3 层叠在一起才是完整的图"——理论上没错。但实际操作中,容器层(扣子)和执行层(Multica)是**不同公司的产品**,打法层(Claude)的 4 种预编排工作流需要**人工设计**。这意味着你要么自己造一个集成层,要么接受"3 个工具各管一段,中间靠人粘"的现实。

### 你漏了一个问题:这 3 家的商业模式冲突吗?

扣子是字节的平台产品,Claude 是 Anthropic 的模型能力,Multica 是独立工具。它们在**生态位上不冲突**,但如果未来有一家开始做"全栈 AI 协作平台",另外两家怎么办?你关注的是技术互补,**商业竞争**可能是另一层需要观察的维度。

### 你自己的笔记流程就是范式的样本

抓文章 2 秒自动化、观察笔记 1:1 复刻、AI 做分析补充、commit + push 自动收尾——你已经活在这套范式里了。只是你自己用的是"人做判断 + AI 做执行"的单人版,还没走到"多 agent 协作"的团队版。
